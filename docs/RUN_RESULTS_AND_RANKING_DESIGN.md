# Концепция итогов забега и рейтингов

Статус документа:
- проектирование
- без кода
- без внедрения
- без привязки к UI-верстке

Цель:
- определить, какую информацию игрок получает после завершения забега
- спроектировать структуру статистики, рангов и комментариев
- подготовить основу для будущего `Экрана результатов`, `Подробной статистики` и `Рейтинга`

## 1. Базовая идея

После каждого завершения забега игрок должен получать не просто сообщение о победе или поражении, а полноценный отчёт о ходе попытки.

Это правило одинаково работает для:
- победы
- поражения
- любого принудительного завершения забега

Итоговая система должна отвечать на 4 вопроса:
1. Что произошло?
2. Как именно игрок зарабатывал и терял очки?
3. Почему забег закончился?
4. Насколько хорошо игрок выступил по сравнению с другими забегами?

## 2. Что игрок видит сразу после завершения забега

### 2.1. Обязательная сводка

Сразу после окончания забега игрок получает компактную сводку:
- итоговое состояние: `Победа` или `Поражение`
- `Очки`
- `Время прохождения`
- `Уничтожено врагов всего`
- `Уничтожено по типам врагов`
- `Босс уничтожен: Да/Нет`
- `Стадия, на которой закончился забег`

### 2.2. Рекомендуемая структура сводки

Верхняя часть:
- статус забега
- итоговые очки
- ранг

Средняя часть:
- время
- общее количество врагов
- статистика по типам врагов
- информация о боссе

Нижняя часть:
- краткий комментарий игры
- кнопки перехода

Все поля, которые нужны для этой сводки, должны поступать из `RUN_ANALYTICS_RUNTIME.md`.

## 3. Разбивка статистики завершённого забега

### 3.1. Общая статистика

Эта группа должна показывать:
- `Очки`
- `Время прохождения`
- `Уничтожено врагов всего`
- `Стадия завершения`
- `Босс уничтожен: Да/Нет`

### 3.2. Статистика по врагам

Отдельно нужно показывать уничтожения по типам:
- `Basic`
- `Tank`
- `Zigzag`
- `Web`
- `Босс`

Если позже появятся другие типы врагов, структура должна быть готова к расширению без смены формата экрана.

### 3.3. Почему эта разбивка важна

Она даёт игроку ответ на вопрос:
- где он набирал очки
- какие враги были самыми опасными
- что именно в забеге пошло хорошо или плохо

## 4. Источники очков

### 4.1. Базовые источники

Система должна явно показывать, сколько очков дают:
- `Basic`
- `Tank`
- `Zigzag`
- `Web`
- `Босс`

### 4.2. Базовая текущая логика как ориентир

Текущий игровой код уже использует такие значения:
- `Basic` - 25
- `Zigzag` - 50
- `Web` - 75
- `Tank` - 100
- уничтожение сегмента босса / ноги - 500

Для будущего дизайна это можно считать стартовой точкой, но сам документ не фиксирует это как окончательную экономику.

### 4.3. Дополнительные источники очков, которые стоит рассмотреть

Кроме убийств врагов, можно учитывать:
- бонус за уничтожение босса
- бонус за чистую победу
- бонус за быстрое прохождение
- бонус за высокий процент сохранённого здоровья
- бонус за уничтожение без полученного урона
- бонус за серию убийств без пауз
- бонус за идеальный забег

### 4.4. Что лучше не делать

Не стоит делать очки слишком размазанными по множеству мелких событий.

Лучше, чтобы игрок понимал:
- за что он получил основную массу очков
- какие действия реально влияют на итог

## 5. Система урона и причины смерти

### 5.1. Зачем она нужна

После поражения игрок должен видеть не только факт проигрыша, но и главные причины, почему забег развалился.

### 5.2. Что показывать после поражения

Минимум:
- `Топ-3 источника полученного урона`

Примеры групп:
- ракеты танков
- паутина
- столкновения
- лазер босса
- spread-атака босса
- ракеты босса

### 5.3. Как должна работать статистика урона

Система должна записывать урон по категориям источника:
- столкновение с врагом
- вражеский снаряд
- опасная зона
- луч
- удар босса
- прочее

Затем на экране поражения сортируются самые опасные категории.

### 5.4. Почему это полезно

Это помогает игроку понять:
- он проиграл из-за ошибок позиционирования
- его наказали конкретные атаки
- проблема была в конкретном типе угрозы

Состав и названия исходных данных для этой статистики должны совпадать с `RUN_ANALYTICS_RUNTIME.md`.

## 6. Подробная статистика

### 6.1. Назначение

Нужен отдельный экран подробностей, который открывается из `Экрана результатов`.

Он не должен заменять краткий итог.

Его задача:
- дать глубже понять забег
- показать аналитику без перегрузки главного экрана

### 6.2. Что можно показывать в подробной статистике

Разбивка очков:
- очки по типам врагов
- очки за босса
- дополнительные бонусы

Разбивка урона:
- урон от врагов
- урон от снарядов
- урон от босса
- урон от опасных зон

Статистика врагов:
- уничтожено каждого типа
- сколько врагов встретилось
- сколько было побеждено без пропусков

Дополнительная аналитика:
- процент точности
- время до босса
- время на босса
- сколько раз игрок был на низком здоровье
- сколько раз игрок получал урон подряд
- максимальная серия уничтожений

### 6.3. Принцип наполнения

Подробный экран должен быть полезным, но не превращаться в стену цифр.

Лучше разделить его на блоки:
1. очки
2. урон
3. враги
4. бонусы
5. аналитика

## 7. Ранги

### 7.1. Зачем нужен ранг

Ранг нужен для мгновенной оценки результата:
- насколько хорошо сыграл игрок
- насколько удачным был забег
- стоит ли повторить попытку ради лучшего результата

### 7.2. Основной кандидат: школьная шкала

Основной кандидат для этой игры:
- `2`
- `3-`
- `3`
- `4-`
- `4`
- `5-`
- `5`
- `5+`

Плюсы:
- русскоязычный формат
- сразу понятен большинству игроков
- хорошо ложится на тон проекта
- легко читается на мобильном экране

Минусы:
- требует аккуратной настройки верхней границы
- промежуточные шаги вроде `3+` и `4+` не очень полезны для аркадного UI и лучше оставить их только если понадобится более тонкая аналитика

### 7.3. Вторичный вариант: словесные звания

Плюсы:
- очень понятный и дружелюбный язык
- хорошо ложится в русскоязычный интерфейс
- создаёт более живой тон

Минусы:
- меньше точности
- сложнее сделать тонкую градацию между соседними уровнями
- длинные подписи хуже читаются в узких UI-блоках

### 7.4. Вариант 3: Редкие авторские звания

Примерная идея:
- нейтральное звание на низких результатах
- более хищные или ироничные звания на высоких

Плюсы:
- сильнее поддерживает характер игры
- лучше подходит для язвительных комментариев

Минусы:
- хуже масштабируется
- сложнее балансировать ощущения
- ранги начинают зависеть от текста, а не от ясной системы

### 7.5. Рекомендация

Для этой игры лучше всего выглядит школьная шкала:
- `2`
- `3-`
- `3`
- `4-`
- `4`
- `5-`
- `5`
- `5+`

Причина:
- игра аркадная
- интерфейс должен быть быстрым для чтения
- русскоязычному игроку проще понимать школьную оценку, чем буквенный код

### 7.6. Текущая игровая шкала как ориентир

Сейчас в коде уже есть базовая шкала по `score`:
- ниже 1500
- ниже 3500
- ниже 6500
- ниже 10000
- выше 10000

Этот диапазон можно считать черновым ориентиром для будущей системы.

## 8. Комментарии игры по итогам забега

### 8.1. Зачем они нужны

Короткие комментарии нужны для:
- эмоциональной реакции
- повторной мотивации
- подчёркивания личного результата
- уменьшения сухости статистики

### 8.2. Формат

Комментарии должны быть:
- короткими
- иногда язвительными
- иногда хвалебными
- не слишком частыми
- зависящими от контекста забега

### 8.3. Категории комментариев

#### Категория 1. Игрок не дошёл до босса
- слишком ранний провал
- игрок потратил попытку на обычных врагах
- игрока быстро выбили из ритма

#### Категория 2. Игрок дошёл до босса, но проиграл
- бой дошёл до главной сцены, но не был закрыт
- игрок был близко, но не справился с паттернами босса

#### Категория 3. Игрок почти победил босса
- босс остался с малым запасом здоровья
- поражение было очень близким
- забег выглядел как почти выигранный

#### Категория 4. Игрок победил с запасом
- победа была уверенной
- здоровья / времени / темпа хватило с хорошим запасом

#### Категория 5. Игрок поставил рекорд
- лучший результат за всё время
- лучший результат по рангу
- лучший результат по скорости
- лучший результат по чистоте прохождения

#### Категория 6. Игрок сыграл чисто
- мало урона
- хороший контроль
- высокий уровень дисциплины

#### Категория 7. Игрок сыграл агрессивно
- много убийств
- высокий темп
- хороший урон в короткое время

### 8.4. Что не нужно делать

Не нужно писать длинные реплики.

Система комментариев должна оставаться:
- короткой
- вариативной
- реактивной
- совместимой с повторными забегами

## 9. Связь с рейтингом

### 9.1. Что должно попадать в рейтинг

Для онлайн-рейтинга лучше сохранять:
- итоговые очки
- ранг
- время прохождения
- факт победы или поражения
- количество уничтоженных врагов
- факт убийства босса
- стадию завершения
- дату и версию забега

Сами поля рейтинга должны собираться из `RUN_ANALYTICS_RUNTIME.md`, чтобы не плодить вторую схему данных.

### 9.2. Что ещё можно сохранять позже

Если потребуется более глубокий рейтинг, можно добавить:
- точность
- полученный урон
- число смертей
- разбор по типам врагов
- лучший комментарий матча

### 9.3. Онлайн-архитектура

Целевой рейтинг должен быть глобальным и онлайн-ориентированным:
- Firebase или аналогичный сервис
- не `localStorage`
- с учётом синхронизации и сетевого ожидания

### 9.4. Какие рейтинговые таблицы возможны

Базовые варианты:
- общий рейтинг по очкам
- рейтинг по скорости прохождения
- рейтинг по чистоте прохождения
- рейтинг по победам

Если система окажется слишком тяжёлой, можно начать с одного общего рейтинга, а потом расширить.

## 10. Предлагаемая структура экранов после завершения забега

### Быстрый экран результатов

Должен показывать:
- итог
- очки
- время
- ранг
- основные источники очков
- топ-3 источника урона, если был проигрыш

### Экран подробностей

Должен показывать:
- разбивку очков
- разбивку урона
- статистику по врагам
- аналитику забега

### Экран рейтинга

Должен показывать:
- глобальный список результатов
- место игрока
- сеть / синхронизацию
- возможность сравнивать свой результат с другими

## 11. Рекомендуемый вывод для первой версии

Для первой версии системы итогов забега лучше всего зафиксировать:

1. Быстрый итог после каждого завершения игры.
2. Подробная статистика как отдельный экран из результатов.
3. Топ-3 причин смерти после поражения.
4. Русскоязычные короткие комментарии игры по категориям.
5. Глобальный онлайн-рейтинг вместо локального хранения.
6. Школьную русскоязычную систему рангов как основной вариант.

## 12. Что надо утвердить до возврата к UI

Перед тем как снова переходить к UI, wireframe и Google Stitch, нужно закрепить:
- форму итоговой сводки
- список статистики
- правило расчёта очков
- правило показа причин смерти
- финальный стиль рангов
- тип комментариев по итогам забега
- состав данных, которые уходят в онлайн-рейтинг
